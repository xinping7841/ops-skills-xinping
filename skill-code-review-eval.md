# 代码评估与审查规范

本规范用于让 Codex/Kun 评估现有代码质量、结构风险、测试缺口和 UI 问题。默认目标是“先审查，不改代码”，除非用户明确要求修复。

## 何时使用

当任务涉及以下内容时，先读本文件：

- 评估项目代码质量、可维护性、可读性、复杂度。
- 做 code review、PR review、风险审查、重构前体检。
- 找 bug、坏味道、重复代码、过度抽象、目录结构混乱。
- 检查测试覆盖、类型安全、错误处理、性能或安全风险。
- 对网页项目做 UI/UX、可访问性、响应式和状态完整性评估。

如果用户说“帮我优化/修复”，先判断是否需要审查报告；若风险较高，先给评估，再小步修复。

## 审查原则

- 先找真实风险，再谈风格偏好。
- 优先报告会导致 bug、数据错误、安全问题、性能退化、维护困难的问题。
- 不因为“可以更优雅”就要求大改；建议必须有收益和风险解释。
- 不把格式化、命名偏好、个人审美放在高优先级。
- 不自动修复，除非用户明确要求“直接改”。
- 避免大而空的总结。每条问题要能定位到文件或模块。

## 严重级别

使用 P0/P1/P2/P3：

| 级别 | 含义 |
|------|------|
| P0 | 会造成生产事故、数据损坏、严重安全问题、核心功能不可用 |
| P1 | 高概率用户可见 bug、关键流程失败、明显性能/稳定性风险 |
| P2 | 可维护性、可测试性、结构边界或中等风险问题，建议近期修 |
| P3 | 小问题、清理项、低风险一致性问题，可顺手处理 |

没有证据时不要升高等级。

## 评估维度

### 1. 正确性

- 核心流程是否符合需求。
- 边界条件、空数据、异常数据、并发或重复提交是否处理。
- 状态流转是否可能进入不一致状态。

### 2. 结构与边界

- 路由、页面、组件、业务逻辑、数据访问是否混在一起。
- 模块是否有清楚职责。
- 是否存在全局状态滥用、循环依赖、跨层调用。

### 3. 可读性

- 命名是否清楚。
- 函数和组件是否过长。
- 条件分支是否过深。
- 注释是否解释原因，而不是掩盖混乱。

### 4. 复杂度与重复

- 是否有明显重复逻辑。
- 是否有过度抽象、万能组件、神对象、巨型 hook。
- 是否可以通过小步提取降低认知负担。

### 5. 类型与数据边界

- TypeScript 是否滥用 `any`、断言或隐式 `unknown`。
- 外部输入、API 响应、表单数据是否校验。
- 日期、金额、ID、枚举、权限等关键数据是否建模清楚。

### 6. 错误处理与用户反馈

- 加载、空状态、错误状态、成功状态是否完整。
- 错误是否可恢复，文案是否说明下一步。
- 日志是否有诊断价值，是否泄露敏感信息。

### 7. 测试与验证

- 是否有 lint、typecheck、unit、integration、e2e、build 检查。
- 关键业务路径是否有测试。
- 测试是否真的断言行为，而不是只跑通快照。

### 8. 安全与隐私

- token、密钥、`.env` 是否误提交。
- 权限检查是否只在前端做。
- XSS、注入、开放重定向、文件上传、敏感日志风险。

### 9. 性能与可靠性

- 是否有不必要的重复请求、巨型 bundle、阻塞渲染。
- 是否缺少分页、缓存、超时、重试、取消请求。
- 是否存在内存泄漏或未清理副作用。

### 10. UI/UX（网页项目）

- 页面是否可扫描，层级是否清楚。
- 桌面和移动端是否正常。
- 可访问性、键盘焦点、对比度、按钮状态是否完整。
- 是否存在文本溢出、布局跳动、卡片套卡片、AI 味装饰。

## 审查流程

1. 读 `AGENTS.md`、`package.json`、路由目录、核心模块、测试脚本。
2. 确认用户要“只审查”还是“审查并修复”。不明确时默认只审查。
3. 优先用 `rg` 搜索风险信号：`TODO`、`FIXME`、`any`、`as unknown`、`console.error`、`dangerouslySetInnerHTML`、`localStorage`、`fetch`、`useEffect`、`eslint-disable`。
4. 阅读高风险路径：登录、权限、支付/金额、删除/发布、表单提交、数据导入导出。
5. 运行可用检查命令；不能运行时说明原因。
6. 输出按严重级别排序的问题清单。
7. 如果用户要求修复，先修 P0/P1，再修 P2，P3 只在顺手且低风险时处理。

## 输出格式

默认输出：

```text
发现 N 个问题：P0 x 个，P1 x 个，P2 x 个，P3 x 个。

1. [P1] 标题
   文件：path/to/file.ts:123
   问题：发生了什么。
   风险：为什么会影响用户或维护。
   建议：最小修复方式。

开放问题：
- ...

验证：
- 已运行 npm run lint：通过/失败
- 未运行 npm run build：原因
```

如果没有发现问题，明确说“未发现阻塞性问题”，并说明剩余风险和测试缺口。

## 新手使用模板

```text
请按代码评估与审查规范检查这个项目。先只审查，不要改代码。重点看正确性、结构边界、可读性、重复代码、类型安全、错误处理、测试覆盖、安全风险和 UI 质量。请按 P0/P1/P2/P3 输出，每条给文件位置、风险和建议修法。
```

网页项目可以加：

```text
同时结合 web-app-dev 技能，检查 React/Next/Vite/Tailwind/shadcn 的目录边界、组件拆分、状态完整性和桌面/移动端 UI 问题。
```

## 与 web-app-dev 的分工

- `web-app-dev`：负责搭建、实现、优化网页服务。
- `code-review-eval`：负责审查、打分、找风险、给修复顺序。

需要修复时，先用 `code-review-eval` 找出优先级，再用 `web-app-dev` 小步实施网页相关修复。
